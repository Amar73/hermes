# MARKO Lab — Paperclip company design (Phase 1: skeleton + code)

Дата: 2026-07-09
Сервер: `193.228.139.46` (alias `hermes`), Paperclip instance `default`

## Контекст

Раздел 10 исходного плана (`2026-06-30-hermes-server-deployment-plan.md`) описывал
мультиагентный workflow для написания Python Telegram-ботов с ролями
Writer/Reviewer/Auditor, привязанными к Pro-подпискам (Claude Code / Gemini CLI /
Codex через OAuth). 2026-07-09 пользователь решил не логинить эти Pro-подписки на
headless-сервере из-за риска бана аккаунта за автоматизацию — см. заметку в шаге 6
плана. Это отменяет исходную схему ролей. OpenRouter (API-ключ, оплата по
использованию, не персональная подписка) остаётся единственным доступным
провайдером моделей.

Отдельно пользователь обозначил более широкую цель для Paperclip-компании: код +
маркетинг/исследования + создание media-контента (текст/посты, видеосценарии,
генерация изображений, генерация видео/аудио). Это явно многосоставная задача —
решено спроектировать и внедрять по фазам. Этот документ — **Фаза 1**: только
костяк компании и роль для кода. Маркетинг/исследования/media — отдельная фаза,
будет спроектирована отдельным заходом после того, как Фаза 1 реально заработает.

Онбординг Paperclip (`paperclipai onboard -y`, выполненный при установке стека)
уже создал одну компанию (id `72c6d782-a89c-404a-a386-cc299a77cff4`) с одним
агентом в роли `chief-of-staff`. Эта компания и агент — основа, не создаём с нуля.

## Область (Фаза 1)

**В скоуп входит:**
- Переименование существующей компании в **MARKO Lab**.
- Существующий агент `chief-of-staff` становится руководителем компании как есть
  (не дублируется отдельным CEO-агентом).
- Один новый агент — **Engineer** (универсал, без разделения на
  Writer/Reviewer/Auditor).
- Поток: задача через Telegram → `chief-of-staff` делегирует → Engineer пишет
  код → коммит в `Amar73/hermes-scripts` → отчёт с ссылкой на коммит обратно в
  Telegram.

**Явно вне скоупа (Фаза 2, отдельный дизайн позже):**
- Отдельный агент «руководитель проектов» (PM) — при двух агентах это лишнее
  звено; добавляется, когда появится второе направление (маркетинг/media),
  которое реально нужно координировать с кодом.
- Roles Reviewer / Auditor / Researcher — были завязаны на Pro-подписки
  (сейчас недоступны) либо на Perplexity (не является нативным провайдером ни в
  Hermes Agent, ни в Paperclip — см. заметку в шаге 10.2 плана). Схема для
  независимой проверки кода будет пересмотрена отдельно, не изобретается здесь.
- Маркетинг/исследования и все 4 типа media-контента (текст/посты,
  видеосценарии, генерация изображений, генерация видео/аудио) — требуют
  дополнительных API-ключей (image/video/audio-генерация), которых пока нет.

## Роли и модели

| Роль | Агент | Модель / провайдер | Задача |
|---|---|---|---|
| Руководитель компании | `chief-of-staff` (существующий) | — (координация, не пишет код) | Принимает задачу из Telegram, делегирует Engineer, формирует финальный отчёт |
| Engineer | новый агент | `anthropic/claude-sonnet-5` через OpenRouter | Пишет Python-код (Telegram-боты), коммитит результат |

Модель для Engineer выбрана намеренно как `anthropic/claude-sonnet-5` — тот же
провайдер и та же модель, что уже настроена, оплачена и проверена end-to-end для
самого Hermes Agent (см. шаг 5 плана). Никакой новой инфраструктуры для Фазы 1 не
требуется.

## Поток задачи (Фаза 1)

1. Пользователь пишет боту в Telegram: «напиши Telegram-бота на Python, который
   делает X».
2. Сообщение попадает `chief-of-staff` (через существующий Hermes-гейтвей,
   уже настроенный и проверенный).
3. `chief-of-staff` делегирует задачу агенту Engineer.
4. Engineer пишет код, коммитит в `Amar73/hermes-scripts` (репозиторий ещё не
   создан — соответствует шагу 10.4 исходного плана, доступ на пуш для
   `paperclip` нужно настроить отдельно).
5. `chief-of-staff` отправляет в Telegram финальный отчёт: что сделано + ссылка
   на коммит.

Без ручных остановок между шагами 2–5 (автономный пайплайн, как и было решено
изначально в разделе 10 плана).

## Что намеренно не решается в этом документе

- Точный механизм создания агента/роли внутри Paperclip (дашборд vs API vs
  конфиг-файл) — неизвестен заранее, будет выяснен по факту при выполнении плана
  через реальный UI/API Paperclip, не изобретается здесь.
- Схема независимой проверки/аудита кода (Reviewer/Auditor) — отложена, см.
  «Явно вне скоупа» выше.
- Дизайн Фазы 2 (маркетинг/исследования/media) — отдельный документ позже.
