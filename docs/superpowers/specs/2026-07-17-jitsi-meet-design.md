# Дизайн: Jitsi Meet на `ovh` (meet.marko-lab.com:50443)

Дата: 2026-07-17
Сервер: `ovh` (51.75.66.60, `ssh ovh`, пользователь `amar`) — тот же, где развёрнут стек Hermes (см. `2026-07-17-ovh-server-deployment-design.md`)
URL для участников: `https://meet.marko-lab.com:50443` (DNS A-запись `meet.marko-lab.com` → `51.75.66.60` уже стоит, проверено)

## Решения пользователя (зафиксированы)

- **Порт 50443, а не 443**: 443 остаётся закрыт файрволом для всех, кроме 5 доверенных IP (за ним живёт дашборд Hermes). Jitsi доступен всем на нестандартном порту — участники получают ссылку вида `https://meet.marko-lab.com:50443/ИмяКомнаты`. Осознанный компромисс: часть гостей за строгими корпоративными файрволами может не пробиться на нестандартный порт.
- **Secure domain**: создавать комнаты может только зарегистрированный пользователь (логин/пароль); гости заходят в существующую комнату по ссылке без пароля, но обязаны представиться (обязательное отображаемое имя). Первый пользователь — `amar`; позже будут заводиться дополнительные (команда добавления должна быть задокументирована для самостоятельного использования).

## Архитектура

Официальный **docker-jitsi-meet** (стабильный релиз), compose-проект в `/opt/jitsi-meet`, всё за существующим Caddy:

| Контейнер | Роль | Сеть |
|---|---|---|
| `web` | веб-интерфейс + config.js | `127.0.0.1:8000` (HTTP, только loopback) |
| `prosody` | XMPP, внутренние аккаунты пользователей | только внутренняя docker-сеть |
| `jicofo` | оркестрация конференций | только внутренняя docker-сеть |
| `jvb` | видеомост (медиа-трафик) | `10000/udp` опубликован на хосте, анонсирует `51.75.66.60` |

Ключевые переменные `.env`: `PUBLIC_URL=https://meet.marko-lab.com:50443`, `JVB_ADVERTISE_IPS=51.75.66.60` (или `DOCKER_HOST_ADDRESS` — по фактическому имени переменной в выбранном релизе), `ENABLE_AUTH=1`, `AUTH_TYPE=internal`, `ENABLE_GUESTS=1`, сгенерированные секреты компонентов (генератор `gen-passwords.sh` из состава релиза).

Причина выбора Docker, а не нативных deb-пакетов: пакет `jitsi-meet` жёстко тянет nginx и рассчитан на владение портами 80/443 — конфликт с уже работающим Caddy; docker-вариант изолирован, конфигурируется через `.env` и обновляется `docker compose pull && docker compose up -d`.

**Docker Engine** ставится из официального apt-репозитория `download.docker.com` (пакеты `docker-ce`, `docker-compose-plugin`). Контейнеры — `restart: unless-stopped` (штатно в компоузе релиза), переживают перезагрузку сервера.

## Caddy

Дополнительный блок в `/etc/caddy/Caddyfile`:

```
https://meet.marko-lab.com:50443 {
    reverse_proxy 127.0.0.1:8000
}
```

Сертификат Let's Encrypt для `meet.marko-lab.com` Caddy выпустит через HTTP-01 challenge на уже открытом порту 80 — нестандартный порт сайта этому не мешает. Вебсокеты (`/xmpp-websocket`, colibri) Caddy проксирует из коробки. Существующий блок `hermes.marko-lab.com` не трогается.

## Файрвол (ufw)

Два новых правила, открытых всем:
- `50443/tcp` — веб-интерфейс Jitsi (через Caddy);
- `10000/udp` — медиа-трафик JVB (WebRTC идёт напрямую на JVB мимо Caddy).

Всё остальное — без изменений (22/443 — только 5 IP, 80 — всем).

## Обязательное имя гостя

`requireDisplayName: true` в config.js. **Точный механизм в docker-версии выяснить по факту** (кандидаты: переменная окружения релиза или кастомный `custom-config.js` в конфиг-томе `web`-контейнера) — не изобретать заранее, проверить на установленной версии; найденный способ записать в раздел «Реально сделано».

## Управление пользователями

Добавление (выполняется на сервере, документируется для самостоятельного использования):

```bash
cd /opt/jitsi-meet && sudo docker compose exec prosody prosodyctl --config /config/prosody.cfg.lua register <логин> meet.jitsi '<пароль>'
```

(точный внутренний домен — `meet.jitsi` по умолчанию в docker-jitsi-meet — сверить с фактическим `.env`). Удаление — `unregister`, смена пароля — `passwd` тем же путём. Первым регистрируется `amar` с паролем, который пользователь сохранит в менеджере паролей.

## Процессные требования

- Для установочных шагов нужен временный passwordless sudo: файл `/etc/sudoers.d/zzz-amar-temp` (именно `zzz-`, см. sudoers-ловушку в спеке основного развёртывания), снять по завершении.
- По завершении — прогнать чек-лист, дописать «Реально сделано», закоммитить и запушить в `Amar73/hermes`, `git pull` на сервере.

## Вне скоупа

- Jibri (запись/стриминг конференций), Etherpad, JWT-аутентификация, TURN-сервер (coturn) — не ставим; если гости за строгими NAT будут испытывать проблемы с медиа, TURN можно добавить отдельным решением позже.
- Перенос Jitsi на порт 443 / отдельный сервер.
